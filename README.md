```
          │               │                │
          ▼               ▼                ▼
   graph_nodes.json  PostgreSQL DB     Groq LLM API
   graph_edges.json  (19 tables)    (llama-3.3-70b)
```

### Why this architecture?

**PostgreSQL** was chosen over SQLite or a dedicated graph database (Neo4j) for three reasons:
1. The dataset is relational at its core — billing items join to sales order items via foreign keys, which SQL handles natively.
2. The LLM-to-SQL path is well-understood and debuggable. SQL is human-readable so you can inspect the LLM's generated queries.
3. Free-tier hosting (Render, Railway) has good PostgreSQL support. Neo4j's free tier is memory-constrained.

The **graph** is a pre-computed JSON overlay (`graph_nodes.json`, `graph_edges.json`) generated by `ingest.py`. This means graph traversal is in-memory and instant — no graph DB needed for the traversal queries in this dataset size.

**Groq** (llama-3.3-70b) was chosen as the LLM provider because:
- Free tier with generous rate limits
- Fastest inference of any free provider (~200 tokens/sec)
- llama-3.3-70b has strong SQL generation capability

## Database Schema

19 tables covering the full O2C flow:

| Layer | Tables |
|---|---|
| Master Data | `customers`, `business_partner_addresses`, `products`, `product_descriptions`, `product_plants`, `plants` |
| Sales | `sales_order_headers`, `sales_order_items`, `sales_order_schedule_lines` |
| Delivery | `outbound_delivery_headers`, `outbound_delivery_items` |
| Billing | `billing_document_headers`, `billing_document_items`, `billing_document_cancellations` |
| Finance | `journal_entries`, `payments` |
| Assignments | `customer_company_assignments`, `customer_sales_area_assignments`, `product_storage_locations` |

Key join paths:
```sql
-- Customer → Orders → Delivery → Billing → Payment
customers.customer
  → sales_order_headers.sold_to_party
  → sales_order_items.sales_order
  → outbound_delivery_items.reference_sd_document  (+ reference_sd_doc_item)
  → billing_document_items.reference_sd_document   (+ reference_sd_doc_item)
  → billing_document_headers.accounting_document
  → journal_entries.accounting_document
  → payments.invoice_reference
```

## LLM Prompting Strategy

### Two-stage pipeline

**Stage 1 — Guardrail** (`isAllowed`):
A lightweight single-call classification that checks if the user's message relates to O2C business data. Returns `ALLOWED | [Module]` or `BLOCKED`. Deliberately permissive — any business analytics question is allowed, only general knowledge / creative writing is blocked.

**Stage 2 — Intent routing** (before SQL generation):
```
message → detectIntent()
  ├── isBroken  → findBrokenOrders() [in-memory graph scan]
  ├── isTrace   → traceGraph(id, depth=4) [depth-limited DFS]
  └── default   → generateSQL() → db.query() → streamAnswer()
```

**SQL generation prompt design:**
- Full schema with column names and data types is injected dynamically at startup
- 2 sample rows per table are included so the LLM understands the data format
- Explicit JOIN HINTS enumerate the foreign-key relationships (the LLM cannot infer these from column names alone)
- Auto-retry up to 2 times on SQL error, with the error message fed back into the prompt

**Answer generation:**
- Streaming via Groq's streaming API for responsive UX
- Last 10 turns of conversation history are included for memory
- LLM is instructed to summarise in 2-3 sentences — the full data is already shown in the table

---

## Guardrails

The system blocks:
- General knowledge questions ("What is the capital of France?")
- Creative writing requests
- Coding help
- Any topic not related to O2C business data

Guardrail implementation uses the LLM itself (fast, flexible) rather than a keyword list (brittle). The prompt is deliberately written to be inclusive of business analytics questions while blocking clearly off-topic ones.

Example blocked response:
> "This system only answers questions about the SAP Order-to-Cash dataset. Please ask about orders, deliveries, billing, payments, customers, or products."

## Bonus Features Implemented

| Feature | Implementation |
|---|---|
| ✅ Natural language → SQL | Dynamic SQL generation with schema injection + auto-retry |
| ✅ Streaming responses | Groq streaming API, chunk-by-chunk WebSocket delivery |
| ✅ Conversation memory | Last 10 turns sent with every message |
| ✅ Graph node highlighting | Query results → node IDs → yellow highlight on graph |

| ✅ SQL query visibility | Collapsible SQL block so users can inspect the generated query |
| ✅ Graph legend | Collapsible legend with entity colors and highlight meanings |

## Setup

### Prerequisites
- Node.js 18+
- Python 3.10+
- PostgreSQL 14+

### Backend
```bash
# Install dependencies
npm install

# Create .env file
cp .env.example .env
# Fill in: DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, GROQ_API_KEY

# Run ingest (one-time)
pip install psycopg2-binary python-dotenv
python ingest.py

# Start server
node server.js
```

### Frontend
```bash
cd frontend
npm install

# Create .env file
echo "VITE_API_URL=http://localhost:3001" > .env
echo "VITE_WS_URL=ws://localhost:3001/ws/chat" >> .env

npm run dev
```

### Environment Variables

| Variable | Description |
|---|---|
| `DB_HOST` | PostgreSQL host |
| `DB_PORT` | PostgreSQL port (default: 5432) |
| `DB_NAME` | Database name (default: sap_o2c) |
| `DB_USER` | Database user |
| `DB_PASSWORD` | Database password |
| `DB_SSL` | Set to `true` for cloud DBs |
| `GROQ_API_KEY` | Groq API key (free at console.groq.com) |
| `PORT` | Server port (default: 3001) |
| `VITE_API_URL` | Frontend → backend HTTP URL |
| `VITE_WS_URL` | Frontend → backend WebSocket URL |
